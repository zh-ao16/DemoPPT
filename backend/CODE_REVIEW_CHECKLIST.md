# DemoPPT 后端代码审查清单

> 每次代码修改后必须逐项检查，通过后才能合并

## 1. API接口审查

### 1.1 请求参数
- [ ] 所有请求参数都有类型定义（Pydantic/注解）
- [ ] 必填参数有 `@Field(..., min_length=1)` 或 equivalent
- [ ] 参数有合理的默认值
- [ ] 请求体使用 `json=...` 而非 query string（登录注册等）

### 1.2 响应格式
- [ ] 成功响应： `{ "success": true, "data": {...} }`
- [ ] 失败响应： `{ "success": false, "error": "错误描述" }`
- [ ] HTTP状态码正确（200成功/400参数错误/401未登录/500服务器错误）
- [ ] 响应字段有驼峰命名规范

### 1.3 认证授权
- [ ] 需要登录的API有 `Depends(get_user_from_header)`
- [ ] Token验证失败返回401而非200
- [ ] 用户ID正确提取（从Header而非硬编码）

---

## 2. 业务逻辑审查

### 2.1 核心流程
- [ ] 用户注册 → 密码加密存储
- [ ] 用户登录 → Token生成且有时效
- [ ] 付费订阅 → 套餐生效
- [ ] AI生成 → 有超时处理、有错误重试

### 2.2 数据一致性
- [ ] 数据库操作有事务控制
- [ ] 多表操作失败能回滚
- [ ] 并发场景有锁机制

### 2.3 边界情况
- [ ] 空输入有处理
- [ ] 超长输入有截断或拒绝
- [ ] 特殊字符有转义（防注入）
- [ ] 超出范围的值有范围检查

---

## 3. 性能审查

### 3.1 API性能
- [ ] AI调用有15秒超时（不含模型排队）
- [ ] 无效API Key立即跳过，不等待60秒
- [ ] 同步函数正确使用（async函数不调用同步阻塞代码）

### 3.2 资源管理
- [ ] 数据库连接用完关闭
- [ ] 大文件不上传Git
- [ ] 日志不打印敏感信息

---

## 4. 安全审查

### 4.1 认证安全
- [ ] 密码不存明文（bcrypt）
- [ ] Token有时效
- [ ] 不能通过遍历ID获取他人数据

### 4.2 数据安全
- [ ] 脱敏存储（masked key如 `sk-9fd...6bd0`）
- [ ] 日志不打印密码/Token
- [ ] SQL注入防护（参数化查询）

### 4.3 权限控制
- [ ] 用户只能操作自己的数据
- [ ] 管理功能有额外权限校验

---

## 5. 代码质量

### 5.1 代码规范
- [ ] 函数有docstring说明
- [ ] 错误有明确日志
- [ ] 无硬编码魔法数字
- [ ] 变量命名有意义

### 5.2 测试覆盖
- [ ] 核心API有单元测试
- [ ] 测试用例有断言
- [ ] 测试通过后才能提交

---

## 6. 影响分析（修改代码前必做）

### 修改XX模块时，检查是否影响：
```
1. 哪些文件调用了这个函数？
2. 哪些API依赖这个数据模型？
3. 前端是否有地方用到这些字段？
4. 数据库迁移是否需要？
```

## 审查流程

```
1. 开发者自检（逐项打勾）
2. 影响分析（列出所有相关文件）
3. 提交代码
4. 自动测试运行
5. 总经理/技术负责人Review
6. 合并
```
